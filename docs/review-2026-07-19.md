# 架构审阅与改进记录（2026-07-19）

> 触发：一份外部模型对 `TsiaohanWang/tau-rs` 的架构审阅建议（覆盖依赖分层、God Crate、错误处理、Provider 抽象、Session 一致性、Compaction、测试深度、TUI 隔离等）。
> 方法：批判性取舍——对照本仓库**实际代码**与原版 `huggingface/tau` 实现，验证建议前提是否成立，再决定采纳 / 暂缓 / 拒绝，并落地高置信、低风险的改进。
> 结论速览：建议整体质量中上，但部分前提与现状不符（如"库 crate 存在 `Box<dyn Error>`"、"tau-coding 是极端 God Crate"）。**本次落地 P0 两条安全/健壮性改进**，其余记录为已知项、暂缓或拒绝，避免无收益的大规模重构。

## 一、逐条批判性评估

### 1.1 依赖反转与分层（建议 1.1）— ✅ 认可，无需改动
现状核实：五层 crate 分离（`tau-types` → `tau-agent` 拥有 `ModelProvider` → `tau-ai` 实现 HTTP → `tau-coding` 组合根 → `tau-cli`）已成立，且 `tau-agent` 零 HTTP 依赖。本仓库维持原样。

### 1.2 Pull-based Stream 语义（建议 1.2）— ✅ 认可，无需改动
`session.prompt()` 返回 `BoxStream`，drop = cancel，已在 TUI 中验证（`run_stream` 通过 `tokio::select!` 驱动）。维持。

### 1.3 Wire 兼容纪律（建议 1.3）— ✅ 认可，无需改动
`deny_unknown_fields` + camelCase + `skip_serializing_if` 已在 `tau-types` 落实。

### 1.4 ADR 治理（建议 1.4）— ✅ 认可
`docs/architecture.md` 与 phase-*.md 已记录 ADR-1~7。

---

### 2.1 🔴 tau-coding 职责过载（God Crate）— ⚠️ 部分认可，但**本次拒绝拆分**
- 现状核实：`tau-coding` 共 **4233 行**，确实承担工具/存储/管理/组合根/compaction/catalog/prompt 多职责。但：
  1. 原版 `huggingface/tau` 的 `tui/` 等本就是单体 Python 模块；Rust 重写时按"coding 组合根"聚合是合理语义边界，并非无约束的 God Crate。
  2. 4233 行在 Rust 重写项目中属中等规模；建议的 `tau-tools` + `tau-session` + 瘦 `tau-coding` 三 crate 拆分将引入至少 3 个新 `Cargo.toml`、workspace 成员、跨 crate 可见性重构，并破坏现有 **99 个** `tau-coding` 单测的路径引用。
  3. 收益（编译解耦）当前不明显：工具 diff 算法改动触发 session 重编译的代价极低（增量编译）。
- **决定**：记录为"已知、暂缓至 Phase 8 评估"，不在本次执行大规模 crate 拆分。若未来工具数量显著增长再议。

### 2.2 🔴 统一错误层级（建议 `TauError` 枚举）— ❌ 拒绝
- 现状核实：经 `grep`，**库 crate（`tau-types/tau-agent/tau-ai/tau-coding`）中不存在 `Box<dyn Error>`**。错误已通过 `thiserror` 分层定义：`SessionError`(storage)、`HarnessError`、`ToolError`、`SessionTreeError`、`JsonlError` 等各自独立；二进制层 `tau-cli` 用 `anyhow` 做 user-facing 渲染（这是 `anyhow` 的标准用法，非缺陷）。
- 建议的"单一 `TauError` 统一枚举"反而**破坏分层**：`tau-types`（纯 serde、零依赖）若定义 `TauError` 并 `#[from] SessionError`，则 `tau-types` 反向依赖 `tau-coding` 的错误类型，违反依赖方向。
- **决定**：拒绝。现有错误分层已是正确设计。补充原则记录："库 crate 用 `thiserror` 各自定义错误，二进制层用 `anyhow` 聚合渲染"——本仓库已遵守。

### 2.3 🟡 Provider 抽象扩展性（建议 `ProviderCapabilities` / `ensure_auth` / `rate_limit_state`）— ⏸ 暂缓至 Phase 8
- 现状核实：`ModelProvider::stream_response` 返回 `BoxStream<'_, AssistantMessageEvent>`，`StreamRequest` 已含 `thinking_level`（Phase 6 落地，由 provider 翻译为 `reasoning_effort` / Anthropic adaptive）。
- `async fn in trait` 现状：返回 `BoxStream` 是 pull-based 语义所需（见 1.2），并非"未用稳定特性"的遗留。
- capability negotiation、OAuth refresh、rate limiter 持久化确为 Phase 8（多 provider）前置。
- **决定**：暂缓。本次不引入 `ProviderCapabilities`（避免为尚不存在的 provider 做过度抽象），Phase 8 实现具体 provider 时再按实际需求设计 capability 接口（YAGNI）。

### 2.4 🟡 Session 持久化并发/一致性（建议 DashMap 分片 / fsync / 文件锁）— ⏸ 暂缓
- 现状核实：`JsonlSessionStorage` 用 `tokio::sync::Mutex` 保护追加写，append-only JSONL，读取时 v1 migration。
- 评估：多进程竞争（TUI + CLI 同时操作同 session）是**理论风险**——当前使用模型是单进程。DashMap 分片与 `fsync` 会增加复杂度且对单进程无收益；跨进程文件锁（`fs2`）可加但属 Phase 8 多进程考量。
- **决定**：暂缓。记录为 Phase 8 候选。

### 2.5 🟡 Compaction 策略原始（chars/4 启发式）— ⏸ 暂缓
- 现状核实：`context_window.rs` 用 `total_chars / CHARS_PER_TOKEN(=4)` 估算 token；`needs_compaction` 为阈值检查；compaction 经 LLM 摘要（三触发）。
- 评估：这是**已知的简化**，非架构缺陷。真实 tokenizer（tiktoken-rs）会引入新依赖且各 provider 不同；分层 compaction 是增强项。
- **决定**：暂缓。维持现状，Phase 8 视真实需求升级。

### 2.6 🟡 测试架构（proptest / fuzzing）— 🟢 **部分采纳（轻量）**
- 现状核实：190 测试（unit + integration），含双向兼容 golden；但无 property-based / fuzzing。
- 评估：SSE parser（`tau-ai/src/sse.rs`）与 JSONL deserializer 处理**不可信外部输入**，确实适合性质测试。
- **决定**：本次为 `sse.rs` 增加轻量 `proptest` roundtrip 性质测试（验证"任意输入不 panic + 确定性"），**不引入独立 `fuzz/` crate 基础设施**（P0 期过重，且与现有 `cargo test` 工作流割裂）。fuzz target 作为 Phase 8 候选记录。

### 2.7 🟢 TUI 隔离性（建议引入 `TuiCommand`）— ✅ 认可方向，本次不重构
- 现状核实：Phase 7 TUI 仅依赖 `tau-types` 事件 + `CodingSession` 只读接口（经克隆 `AgentHarness` 句柄做 steer/cancel），已满足隔离原则。
- `TuiCommand` 命令模式在 TUI 功能增长（inline diff、file tree）时确有价值，但当前 TUI 为最小可用子集，引入命令模式属过度设计。
- **决定**：记录为 TUI 演进时的架构约束（"TUI 只产生意图，由 session 执行"），本次不落地。

## 二、本次落地的改进（P0）

### P0-a：安全底线 —— 全局 `deny(unsafe_code)` + 必要 unsafe 局部豁免
- **动机**：本仓库执行任意 shell 命令（`bash` 工具），是攻击面；全局 unsafe 软禁是安全工程底线（建议 P0-1 正确采纳）。实际采用 `#![deny(unsafe_code)]` 而非 `#![forbid(...)]`，因为 `forbid` 无法被子模块 `#[allow]` 局部豁免，而 `deny` + 局部 `allow` 达到相同的"默认禁止、显式豁免"保证。
- **现状 3 处 `unsafe`**：
  1. `tau-cli/src/config.rs:207,212` — 测试内 `env::set_var`/`remove_var`（单线程测试，有 SAFETY 注释）。
  2. `tau-coding/src/tools/bash.rs:69` — `libc::killpg(pid, SIGTERM)` 取消**进程组**（Rust 标准库 `Child::kill` 仅杀单进程，无法杀 shell 派生的子进程，故**必要**）。
- **做法（比建议更精细）**：
  - 在 5 个 crate 根加 `#![deny(unsafe_code)]`。
  - 将 `bash.rs` 的 `libc::killpg` 收敛到**独立小模块** `tau-coding/src/tools/unix_kill.rs`，该模块用 `#![allow(unsafe_code)]` 局部豁免 + 显式 `// SAFETY:` 注释解释为何必须调 libc。其余所有代码（包括 bash.rs 主体）处于全局 deny 之下。
  - `config.rs` 测试的两处 `env::set_var` 用 `#[allow(unsafe_code)]` 局部包住（保持单线程测试语义）。
- **效果**：默认禁止任何 unsafe；唯一 unsafe 被显式隔离、注释、可审计。优于"全局 deny 后不知道哪里还有 unsafe 被悄悄允许"。

### P0-b：SSE parser 性质测试
- 为 `tau-ai/src/sse.rs` 增加 `proptest`：任意字符串输入 `parse_sse_line` 不 panic，且对合法 `data: <json>` 行 roundtrip 稳定。
- 不引入独立 fuzz crate（避免 P0 期基础设施膨胀）。
